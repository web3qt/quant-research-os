<objective>
Add a lightweight installed-wrapper provenance guard so QROS bin entry points refuse or clearly warn when `.qros/install-manifest.json` points to a missing source checkout, a source checkout that violates an explicit `QROS_EXPECTED_SOURCE_REPO`, or a source checkout whose dirty state contradicts the manifest before delegating to session, review, validation, progress, update, and diagnostics scripts.
</objective>

<threat_model>
**Assets:** Runtime bin wrappers, installed `.qros/install-manifest.json`, active research repo trust boundary, QROS source checkout.

**Threats:**
- **High:** Review/session commands execute against a different QROS clone than the maintainer explicitly expects via `QROS_EXPECTED_SOURCE_REPO`, causing false PASS or stale contract behavior.
- **Medium:** A wrapper silently resolves `RUNTIME_ROOT` from manifest without checking manifest/source consistency.
- **Low:** Strict provenance blocking could make recovery harder if `qros-update` itself is blocked.

**Mitigations in this plan:**
- Add wrapper-level provenance checks before script delegation.
- Allow `qros-update` to run in recovery mode while still printing drift diagnostics.
- Add `QROS_ALLOW_PROVENANCE_DRIFT=1` override for emergency/manual recovery, with explicit warning text.
</threat_model>
