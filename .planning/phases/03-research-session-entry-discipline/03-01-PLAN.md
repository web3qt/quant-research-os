# Plan 03-01: Deterministic Stage Entry Guard

**Objective:** Add a runtime-owned admission check for stage-specific author/review skills so they cannot proceed when the lineage current stage belongs to a different phase of `qros-research-session`.

**Wave:** 1
**Requirements:** ENTRY-01, ENTRY-02, ENTRY-03, ENTRY-04

## Tasks

1. Add focused red tests.
   - Runtime allow/block tests for author and review lanes.
   - CLI tests for diagnostic output.
   - Skill guidance tests requiring the guard.
   - Bootstrap/install tests requiring the new wrapper.

2. Implement the guard.
   - Add `runtime/tools/stage_entry_guard.py`.
   - Add `runtime/scripts/check_stage_entry.py`.
   - Add `runtime/bin/qros-check-stage-entry`.
   - Reuse `detect_session_stage`, `STAGE_ACTIVE_SKILLS`, and latest lineage selection.

3. Update skills and docs.
   - Add mandatory runtime stage guard sections to author/review skills.
   - Update `qros-research-session` and usage docs to describe normal progression vs stage-specific admission.

4. Verify.
   - Run focused tests.
   - Run docs/bootstrap minimum.
   - Run `smoke`.
   - Run `full-smoke`.

## Expected Behavior

Allowed:

- `qros-check-stage-entry --stage csf_data_ready --lane author` when current stage is `csf_data_ready_confirmation_pending` or `csf_data_ready_author`.
- `qros-check-stage-entry --stage csf_data_ready --lane review` when current stage is `csf_data_ready_review_confirmation_pending` or `csf_data_ready_review`.

Blocked:

- `qros-check-stage-entry --stage csf_data_ready --lane author` when current stage is `mandate_next_stage_confirmation_pending`.
- `qros-check-stage-entry --stage csf_data_ready --lane review` when current stage is `csf_signal_ready_review_confirmation_pending`.

The blocked result must name:

- Observed `current_stage`.
- Requested `stage` and `lane`.
- Expected runtime stages.
- Current active skill.
- Recovery command through `qros-research-session`.
